# Dan Koe Writer Skill V5.0

> 研究驱动的内容积木系统，学习 Dan Koe 方法论，创作有深度的爆款内容
>
> **核心理念**：先研究再写作，积木全量生成、动态组装。
> 本 Skill 负责结构与内容，Chinese Style Optimizer 负责文笔优化。

---

## 创作流程

```
Phase 1：理解需求
│  Step 1：用户提出主题
│  Step 2：确认身份 / 平台 / 篇幅
│
Phase 2：研究素材
│  Step 3：联网研究（3-6 次搜索）
│
Phase 3：选择方向
│  Step 4：提供 6+ 切入角度（每个角度关联真实数据）
│  Step 5：用户选择角度
│
Phase 4：深度分析
│  Step 6：AI 内部深度追问（用户不可见）
│  Step 7：向用户询问细节（可答可不答）
│
Phase 5：生成积木
│  Step 8：生成完整积木库（8 类，约 32 块）
│  Step 9：强制多样性检查
│
Phase 6：组装输出
│  Step 10：动态匹配，推荐 3 个组装方案（同平台同篇幅，不同 SOP/调性）
│  Step 11：用户选择方案
│  Step 12：生成初稿
│  Step 13：提供 2-3 个标题选项
│
Phase 7：文笔优化
│  Step 14：传递给 Chinese Style Optimizer
│
Phase 8：完成
│  输出最终版本
│  用户可说"再出一个 XX 平台版本" → 回到 Step 10 用同一批积木重新组装
```

---

## 各步骤详细规则

### Step 2：确认基础信息

通过 `ask_user_input` 一次性收集：
- 平台（小红书 / 公众号 / 知乎 / Twitter / Newsletter / 即刻 / 其他）
- 篇幅（短文 500 字以内 / 中文 1500-3000 字 / 长文 3000+ 字）
- 作者身份（一句话描述，如"芳疗师""独立开发者"）
- 可选：立场偏好 / 个人经历 / 目标读者 / 语气偏好

#### 2b. 读者原型推断（AI 自动执行，用户确认）

根据主题 + 平台 + 作者身份，AI 推断 2-3 个**读者原型**：

```
格式：
原型 A：[一句话画像] — [核心痛点] — [阅读动机]
原型 B：[一句话画像] — [核心痛点] — [阅读动机]
原型 C：[一句话画像] — [核心痛点] — [阅读动机]

示例（主题：拖延症，平台：公众号，身份：心理咨询师）：
原型 A：28 岁互联网运营，每天加班但核心项目一直拖 — 怕做了被否定 — 想找到"不靠意志力"的方法
原型 B：35 岁全职妈妈想重返职场但迟迟不投简历 — 怕被拒绝证明自己过时了 — 想要心理松绑
原型 C：大三学生论文拖了两个月 — 完美主义导致无法开始 — 想要立即可用的起步技巧
```

- 展示给用户，用户可以确认、修改或跳过
- 后续 Step 8 生成积木时，每块积木标注它最击中哪个原型（A/B/C/通用）

---

### Step 3：联网研究

**必须使用 `web_search` 和 `web_fetch`**，不可跳过。

**搜索策略（3-6 次，短文快车道 2 次）**：

**严禁只搜索干瘪的科学定义，必须搜索"人"的困境**：

1. **痛点/崩溃场景搜索**：大众在面对这个问题时，最常见的"无效努力"或"情绪崩溃场景"是什么？
   - 搜索词构造规则：**必须包含平台名或社区名**
   - 示例：`小红书 + [主题] + 吐槽/翻车/后悔/踩坑`、`豆瓣 + [主题] + 崩溃/放弃`
   - 提取带情绪词的句子和带数字的句子，这些是最好的积木原料

2. **反面/争议搜索**：寻找与主流常识完全相反，但逻辑自洽的硬核观点。
   - 搜索词构造规则：**必须用反面关键词**
   - 示例：`为什么 [主流建议] 没用`、`[主题] 骗局/误区/真相`、`stop [mainstream advice]`

3. **核心主题现状**：这个领域当前最大的变化或噪音是什么？
   - 搜索最新数据、趋势报告、行业变化

**搜索结果优先提取**：
- ✅ 带具体数字的句子（"73% 的人…"）
- ✅ 带强烈情绪的真实用户发言
- ✅ 反常识的研究结论
- ❌ 百科式定义
- ❌ 泛泛的"专家建议"

输出：编译成内部「高对比度原始素材摘要」。包含：大众谬误、真实痛点场景、反常识数据/理论。

**用户不可见此摘要，但后续所有步骤都基于它。**

### Step 4：提供切入角度

- 提供 **6 个切入角度**（短文快车道提供 3 个）
- **绝对禁止**提供"盘点型"、"分类大全"、"N个方法"这种平庸的百科式角度
- 每个角度必须是一把"利刃"：

```
格式：
[角度名称]
大众认为：___
实际上：___
为什么深刻：___
平台适配度：[高/中/低] — [一句话原因]
```

**平台基因规则**：6 个角度中至少 2 个需要标注平台适配预测：
- 小红书：偏好"个人经历型"和"反焦虑型"角度，读者需要共鸣感
- 公众号：偏好"系统分析型"和"行业洞察型"角度，读者需要深度感
- 知乎：偏好"专业拆解型"和"反常识型"角度，读者需要信服感
- Twitter/X：偏好"一击必杀型"和"争议型"角度，读者需要转发冲动
- 即刻：偏好"个人感悟型"和"行业观察型"角度，读者需要同频共振

极度重要：思维剥离规则】你必须使用 Dan Koe 的思维框架，但绝对禁止照搬他的科技词汇！

这是决定文章质量的最关键一步！AI 在内部执行，严禁使用百科式科普。

**参考 `methods/dankoe-methods.md` 获取底层方法论，但只提取逻辑骨架，必须用用户主题的语言重新表达。**

#### 6a. 强行套用心智模型（只套逻辑，不套词汇）

每个模型必须产出一个**填空式输出**，不可模糊带过：

**【身份-行为模型】**：
```
填空：在 [主题] 上，大众的错误身份认同是「___」。
他们以为自己是「___」，实际上他们在扮演「___」。
所以他们的行为变成了「___」，而这恰恰导致了「___」。
```

**【目标镜头理论】**：
```
填空：大众在 [主题] 上的隐藏目标其实是「___」（比如追求安全感、逃避评判）。
这个隐藏目标扭曲了他们的行动方式：他们以为自己在「___」，
其实他们在「___」。
```

**【系统演化视角】**：（长文必做，短文跳过）
```
填空：如果把 [主题] 放到 [更大的系统] 里看，
5 年前的正确做法是「___」，现在失效了因为「___」，
下一步的演化方向是「___」。
```

#### 6b. 构建"共情式"反共识

**矛头指向规则**：不准指责读者"你做错了"，要把矛头指向"社会常识的错"或"外界压力"，给读者心理松绑。

```
填空：主流解决方法是「___」。
它错得离谱，因为「___」。
越努力越糟糕的机制是：「___」。
打破这个循环的断层在于：「___」。
```

#### 6c. 构建具象的"痛点切片"

**硬规则**：不允许使用抽象人群描述。必须描写具体的**动作 + 时间 + 环境 + 心理独白**。

```
❌ "压力大的人往往睡不好"
❌ "很多职场人感到焦虑"
❌ "创业者面临诸多挑战"

✅ "凌晨 2 点，第 4 次删掉写了一半的辞职信，然后打开外卖 app 点了一份不饿但需要的炸鸡"
✅ "周日晚上 10 点打开工作群，看到 47 条未读，手指悬在屏幕上方，最终锁屏去刷了 40 分钟短视频"
```

至少生成 **3 个痛点切片**，标注它们分别对应哪个读者原型（Step 2b）。

#### 6d. 提取同频的底层类比

放弃高冷的物理/经济学跨界！用生活中最直白、有温度的事物。

**黑名单（严禁使用）**：护肤、健身、常见感冒、电脑/CPU/手机、驾驶汽车、种树/种花
**白名单方向**：厨房里的意外发现、深夜独处的时刻、童年记忆、身体的本能反应、旅行中的错路

#### 6e. 三重质量关卡

**关卡 1 — 维基百科测试**：
```
如果这段分析中超过 40% 的句子以以下词语开头 → 不合格：
"研究表明" / "通常来说" / "一般认为" / "专家指出" / "据调查"
→ 回到 6a 推翻重写
```

**关卡 2 — "所以呢"测试**：
```
对每条核心洞察连续追问 3 次"所以呢？"/"这意味着什么？"
如果第 3 次追问后无法继续深挖 → 说明第 1 层不够深 → 重写
如果第 3 次追问后的答案比第 1 层更有力 → 用第 3 层替换第 1 层
```

**关卡 3 — 过载检测**：
```
如果整段分析读起来像一篇四平八稳的科普文章或产品说明书 → 不合格
它应该听起来像：一个聪明的朋友在深夜跟你说的那种"操，原来是这样"的洞察
```

### Step 7：向用户询问细节

通过 `ask_user_input` 或直接提问，收集个人素材：
- "你有没有相关的个人经历或客户案例？"
- "你的读者最常见的误区是什么？"
- "有没有想特别强调的行动建议？"

**用户可以全部跳过。** 跳过则 AI 用内部分析结果继续。
如果用户补充了细节 → 优先融入故事积木和行动步骤积木。

---

### Step 8：生成完整积木库

**永远全量生成**（短文快车道除外），不因篇幅删减。这是素材池，组装时才做筛选。

| 积木类型 | 数量 | 说明 |
|----------|------|------|
| 核心洞察 | ×3 | 深度观点，文章灵魂 |
| 金句 | ×5 | 短小精炼，传播利器。参考 `patterns/golden-phrases.md` 但必须用中文原生节奏 |
| 故事 | ×3 | 标注来源：个人经历 / 研究案例 / 读者场景 |
| 数据 | ×3 | 必须有真实来源，不可编造 |
| 类比 | ×4 | 跨领域比喻，降低理解门槛。遵守 6d 的黑白名单 |
| 反常识钩子 | ×4 | 打破认知，开场引爆。参考 `patterns/hooks.md` |
| 行动步骤 | ×5 | 具体方法，立即可做 |
| 问题 | ×3 | 引发思考，互动引导 |

**短文快车道积木量**：核心洞察 ×2、金句 ×2、故事 ×1、类比 ×1、反常识钩子 ×1、行动步骤 ×1（共 6-8 块）

积木内容来源：
- 研究素材（Step 3）提供事实基础
- 深度追问（Step 6）提供分析深度
- 用户细节（Step 7）提供个人素材
- Dan Koe 文章分析（knowledge-base/articles/）提供写作模式参考

**积木硬规则**：
- 所有积木必须 100% 严格服务于用户在 Step 5 选择的具体角度！严禁跑题
- 每一块积木只能包含**一个单一概念**
- 绝对禁止在单块积木内部出现列表（如 1234）或多重分支
- 如果概念太大，必须将其切片。短小、锐利、一击必杀
- 每块积木标注**读者原型适配**（A/B/C/通用）

**金句中文化硬规则**：
```
金句必须用中文原生节奏，禁止翻译体。检查标准：

❌ 翻译体："你不在你想去的地方，因为你还不是会在那里的那个人"
✅ 中文原生："不是路太远，是你还在假装赶路"

❌ 翻译体："生产力不是做更多，而是做重要的事"
✅ 中文原生："忙，是另一种形式的逃避"

技巧：四字格对仗、反问收尾、口语化短句、留白让读者自己悟
```

#### Step 8b：积木张力标注（新增）

积木全部生成后，执行张力关系标注：

```
扫描所有积木，找出以下关系对：
- 对抗：积木 A 和积木 B 观点相反，可形成"正反打"
- 递进：积木 A 是积木 B 的铺垫，B 比 A 深一层
- 呼应：积木 A 和积木 B 从不同角度说同一件事
- 反转：积木 A 建立预期，积木 B 打破预期

输出格式：
[洞察#1] ←对抗→ [钩子#3]：读者先被钩子的反常识震住，再被洞察的深度说服
[故事#2] ←递进→ [行动#1]：故事制造共鸣后，行动步骤自然落地
[金句#4] ←反转→ [数据#2]：金句给出直觉判断，数据给出反直觉证据
```

标注至少 **5 对**张力关系。Step 10 组装时优先使用有张力关系的积木组合。

### Step 9：多样性检查

确保积木在同一个角度下，有情绪共鸣（痛点故事）、有微观特写、有宏观机制解释。
严禁出现偏离主题的类比。

```
□ 核心洞察（3 条）：是否探索了不同侧面？
  测试：如果 3 条可以用同一句话概括 → 不通过

□ 金句（5 条）：语气是否有变化？
  要求：至少覆盖 3 种语气（犀利 / 温暖 / 哲学 / 幽默 / 挑衅）
  附加：是否全部通过"中文原生"检查？有翻译腔 → 重写该条

□ 类比（4 条）：是否来自 3 个以上不同领域？
  反例：全是科技比喻（CPU / 电脑 / 手机）→ 不通过
  附加：是否触碰 6d 黑名单？→ 替换

□ 故事（3 条）：是否包含不同类型？
  要求：个人经历 / 研究案例 / 读者场景 至少覆盖 2 种
  附加：是否有具象的痛点切片（动作+时间+环境+心理独白）？无 → 不通过

□ 反常识钩子（4 条）：强度是否有梯度？
  要求：轻微好奇 → 中度挑衅 → 强烈颠覆，不能全是同一强度

□ 行动步骤（5 条）：是否覆盖不同类型？
  要求：立即可做 / 应该避免 / 深度练习 至少覆盖 2 种

□ 读者原型覆盖：3 个原型是否都至少被 3 块积木命中？
  不达标 → 为被忽略的原型补充积木
```

**复读机检测**：
```
提取所有积木的核心关键词，
如果超过 60% 在表达同一个意思 → 判定为"复读机模式"
→ 回到 Step 6 重新做深度追问，寻找更多维度
```

### Step 10：动态匹配组装方案

**原则：积木全量保留，组装时根据平台/篇幅动态挑选。**

#### 10a. 根据篇幅确定组装规则

**短文（500 字以内）**：
- 绝对禁止调用 content-sop.md
- **强制参考 `patterns/short-form.md`** 的规则和结构（结构A/B/C）
- 一击必杀，最多用 2-3 块原子积木，拒绝列表
- 所有短文规则以 `patterns/short-form.md` 为唯一权威来源

**中文（1500-3000 字）**：
- 使用 5-7 块积木
- 可以有完整故事展开
- 可以有多步方法论
- 允许 1 次"对比转变"结构

**长文（3000 字以上）**：
- 积木使用不限
- 可以多线叙事
- 完整展开故事、数据、方法论

#### 10b. 结构选择决策树

根据当前主题 + 平台 + 篇幅，按以下逻辑匹配结构（参考 `patterns/structures.md`）：

```
主题偏认知/心理 → 层层深入型 或 微故事型
主题偏行动/方法 → 问题→分析→方案型
主题偏趋势/时代 → 史诗演化型 或 维度递进型
主题偏经验/复盘 → 规则列表型
主题偏情感/共鸣 → 故事先行型
不确定 → 对比论证型（最通用）
```

再从 `sop/content-sop.md` 中匹配对应的 SOP 模板。

#### 10c. 推荐组装方案

**中长文**：推荐 3 个方案。**短文**：推荐 2 个方案。

每个方案展示：
```
方案 N：[名称]
结构类型：[层层深入型 / 对比论证型 / ...]
SOP 参考：[SOP-001 / SOP-002 / ...]
使用积木：[列出编号]
积木张力线：[标注关键积木之间的衔接逻辑：递进 / 转折 / 呼应 / 对抗]
情绪弧线：[一句话描述]
命中读者原型：[主要命中 A / B / C]
为什么推荐：[简要说明]
```

**方案多样性硬规则**：
- 方案之间必须使用不同的结构类型
- 情绪弧线必须明显不同（读出来像不同人写的）
- 不是简单地调换积木顺序
- 所有方案必须基于用户选择的同一个角度！只是行文结构不同
- 尽量让不同方案命中不同的主要读者原型

---

### Step 12：生成初稿

- 严格遵守 Step 10a 的篇幅组装规则
- 展开积木时自然融入，不机械堆砌
- **优先使用 Step 8b 标注的张力关系来安排积木顺序**
- 参考 `patterns/` 下的钩子、金句、标题模式
- 参考 `methods/dankoe-methods.md` 中的方法论框架（只用逻辑，不用原词）
- 用微故事法（Problem → Amplify → Solution）作为最小写作单元——任何段落都可以从一个相关问题开始

**段落级写作乐高（展开积木时的填充工具）**：

积木系统告诉你"这篇文章由哪些大块组成"。
写作乐高告诉你"这个大块内部怎么展开"。

当展开某块积木、不知道下一句写什么时，循环检查以下选项：

```
□ 痛点 — 用一个相关痛点起头，思路会自然流动
□ 例子 — 扔一个具体例子让观点落地
□ 个人故事 — 3 句话的场景闪回
□ 统计数据 — 找一个真实数字增加权威
□ 隐喻 — 像对小孩解释一样
□ 引用 — 用名言强化观点
□ 重构视角 — 换个角度再看一次
□ 追问为什么 — 问自己 what/how/why，继续深挖
```

**乐高使用规则**：
- 不需要全用，一个段落用 1-2 种就够
- 写到卡住时从上到下过一遍清单，选最自然的
- 短文（500字以内）最多用 2-3 种乐高

**初稿反模式检查（写作过程中持续执行）**：
参考 `patterns/anti-patterns.md`，确保不触碰以下高频错误：
- 禁止使用"在当今社会"、"随着...的发展"、"众所周知"等空洞开头
- 禁止连续 3 个以上段落都以陈述句开头（必须打破节奏）
- 禁止在非学术文章中使用"首先...其次...最后..."
- 禁止在结尾写"总之"、"综上所述"、"希望本文对你有帮助"
- 禁止同一个关键词在 500 字内出现超过 5 次
- 禁止翻译腔的长定语从句（"那些在深夜里独自面对着屏幕上的代码的程序员们"）

### 篇幅适配检查（初稿完成后自动执行）

```
短文检查：
□ 总字数是否在目标范围内？
□ 叙事线是否只有 1 条？
□ 是否出现了"以前 vs 现在"对比段？（不允许）
□ 行动步骤是否超过 3 步？（不允许）
□ 是否有不必要的过渡句 / 总结句？
□ 同一关键词出现是否超过 5 次？（复读机检测）
□ emoji 是否超过 5 个？（小红书可放宽到 8 个）
□ 是否通过 short-form.md 的 4 个特征检查？
   （一个观点打穿 / 节奏感代替论证 / 类比即论证 / 结尾开放）

中长文检查：
□ 总字数是否在目标范围内？
□ 开头 100 字内是否有钩子？（没有 → 重写开头）
□ 是否有至少 1 个痛点切片（具象场景）？
□ 核心洞察是否在文章前 1/3 处出现？（不能藏太深）
□ 结尾是否有力？（金句 / 行动号召 / 开放问题，三选一）
□ 段落长度是否有变化？（全是长段 → 拆分；全是短句 → 合并部分）
```

## 短文快车道详细规则

当 Step 2 确认篇幅为"短文（500 字以内）"时，自动切换到快车道：

| 标准流程 | 快车道 | 原因 |
|----------|--------|------|
| Step 3：3-6 次搜索 | 2 次搜索 | 短文不需要过多素材 |
| Step 4：6 个角度 | 3 个角度 | 减少选择负担 |
| Step 6：全部执行 | 只执行 6a + 6c | 短文没空间展开系统思考和宏观演化 |
| Step 8：32 块积木 | 6-8 块精选积木 | 最终只用 2-3 块，生成 32 块浪费 |
| Step 8b：张力标注 | 跳过 | 积木少，张力关系简单 |
| Step 10：3 个方案 | 2 个方案 | 短文结构选择有限 |

**快车道的 Step 6-lite**：
- 只执行 6a（身份-行为模型 + 目标镜头理论）和 6c（痛点切片）
- 跳过 6b（反共识构建）和 6d（类比提取）——短文中类比直接在积木阶段生成即可
- 6e 质量关卡仍然执行，但只检查关卡 1（维基百科测试）
